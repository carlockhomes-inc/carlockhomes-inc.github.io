システム開発 危機管理ガイド

CarlockHomes

本ガイドは、開発現場で発生するトラブルに対して、チーム全員が冷静かつ迅速に対応するための共通指針です。 「犯人探し」ではなく「解決」に集中する文化を作ります。

1. バグ報告の「4大要素」

トラブルを見つけた際は、以下の項目に沿って報告してください。これにより状況把握が5分短縮されます。

項目 解説 具体例
期待値 本来どう動くのが正しいか 「ログインボタンを押すと管理画面が開く」
現状 何が起きているか（エラー文など） 「ボタンを押すと白い画面のまま止まる」
再現方法 どうすれば同じ現象が起きるか 「iPhoneのSafariからアクセスして実行」
深刻度 優先順位の判断材料 緊急：全停止 普通：一部不具合

2. 「アクセス集中」で落ちる仕組み

💡 例え話：スーパーの特売日

システムが止まるのは、「1つしかないレジに一気に1,000人が並んだ」のと同じ状態です。店員（サーバー）が処理しきれず、行列が長くなりすぎて客が帰ってしまう（エラー）のが原因です。

原因：キャパオーバー

計算能力（CPU）や記憶域（メモリ）がリクエストの数に対して不足したときに起こります。

対策：分散と拡張

レジ自体を増やす（サーバー増強）か、レジを通さず渡せるお弁当（キャッシュ）を用意します。

3. チームの約束（チェックリスト）

  • 課題は GitHub Issues に集約する：チャットで流さず、チケットとして可視化する。
  • 「切り戻し」を計画に入れておく：解決に時間がかかるなら、すぐに前日の状態に戻す。
  • バグ報告を称賛する：大きな爆発の前にバグを見つけた人は、プロジェクトの救世主です。
  • キャパシティを事前に共有する：同時接続 何人まで耐えられる設計か、共通認識を持つ。
