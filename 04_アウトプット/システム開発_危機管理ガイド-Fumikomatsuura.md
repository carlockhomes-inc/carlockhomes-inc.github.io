# システム開発：危機管理と安定運用のためのガイド

このドキュメントは、システム開発における「予期せぬトラブル（バグ）」と「高負荷時の停止」に対して、チームがパニックにならず、冷静に対処するための共通指針です。

---

## 1. バグ発生時に慌てないための「心得とルール」

バグは「悪いこと」ではなく、システムをより良くするための「改善の種」です。

### なぜパニックになるのか？
- 「どこで何が起きているか」が不明確だから。
- 「誰の責任か」を考えてしまうから。
- 「いつ直るのか」が見えないから。

### 対策：バグ報告の「共通フォーマット」
トラブルを発見した際は、感情を横に置き、以下の4点を報告するルールにしましょう。

| 項目 | 内容 | 例 |
| :--- | :--- | :--- |
| **期待値** | 本来どう動くべきか | 「ログインボタンを押すと管理画面が開く」 |
| **現状** | 実際はどうなったか | 「ログインボタンを押すと白い画面のまま止まる」 |
| **再現方法** | どうすれば再現するか | 「ブラウザでiPhoneからアクセスすると発生」 |
| **深刻度** | ビジネスへの影響度 | 「高：全ユーザーが使えない」「低：誤字がある」 |

> **💡 アドバイス**
> 中島聡氏の「ロケットスタート」のように、開発初期にバグを出し切ることは成功への近道です。バグを見つけた人を「発見者」として称賛する文化が、隠蔽を防ぎ、大きな危機を回避します。

## 2. 「アクセス集中で繋がらない」の正体

システムが止まる、あるいは極端に重くなる現象を「サーバー負荷」と呼びます。

### 例え話：1つしかないスーパーのレジ
- **通常時**: 客が1人ずつ来るので、レジ（サーバー）はスムーズに回転します。
- **アクセス集中時**: 特売日に1000人が一気にレジに押し寄せた状態です。
  - **サーバーダウン**: 店員がパニックで倒れてしまう状態。
  - **タイムアウト**: 行列が長すぎて、客が「もう待てない」と帰ってしまう状態。

### なぜ開発中に気づけないのか？
開発中は「店員3人に対し、客が1人」のような贅沢な環境でテストしているからです。実際に「1000人が一気に来る」状況は特殊なテスト（負荷テスト）をしない限り露呈しません。

### アクセス集中への対策（テクニカル）
1. **オートスケーリング**: 客が増えたら自動で「レジ（サーバー）」を増やす設定。
2. **キャッシュ**: 毎回レジを通さなくても、外で配れる「お弁当（静的データ）」を用意しておく。
3. **順番待ち（キュー）**: 溢れた客を「行列」として並ばせ、順番に処理する仕組み。

## 3. チームで守るべき「4つの約束」

1. **GitHub Issues を活用する**:
   バグや課題はチャットで流さず、GitHubの「Issue」にチケットとして起票し、進捗を可視化しましょう。
2. **「切り戻し」を恥じない**:
   バグが解消できない場合、一度「昨日の正常な状態」に戻す勇気を持ってください。
3. **ログを愛する**:
   「動かない」ではなく「このエラーメッセージが出ている」という証拠（ログ）を必ずエンジニアに共有してください。
4. **キャパシティ（容量）を知る**:
   リリース前に「同時接続は何人まで耐えられるか？」をエンジニアに確認しておきましょう。
